---
title: Docs site structure
description: Organize Velu configuration, MDX pages, assets, and API specifications in one project.
---

A Velu docs-as-code project keeps configuration, authored content, and
project-owned assets under one project root. Run Velu commands from the
directory that contains `velu.json`.

## Use a predictable project layout

<Tree>
  <File name="velu.json" />
  <File name="favicon.svg" />
  <File name="index.mdx" />
  <File name="quickstart.mdx" />
  <File name="openapi.json" />
  <Folder name="guides" defaultOpen>
    <File name="configuration.mdx" />
    <File name="deployment.mdx" />
  </Folder>
  <Folder name="images" defaultOpen>
    <File name="dashboard.png" />
    <File name="logo.svg" />
  </Folder>
</Tree>

Directory names are flexible. Navigation and asset references must match the
actual project-relative paths.

## Understand the project files

<Field name="velu.json" type="JSON" required>
Defines the project name, appearance, navigation, API behavior, contextual
actions, and other site-wide settings.
</Field>

<Field name="*.mdx" type="MDX" required>
Contains page frontmatter, Markdown prose, code examples, images, and built-in
Velu components. Navigation resolves authored pages to `.mdx` files.
</Field>

<Field name="openapi.json or openapi.yaml" type="OpenAPI">
Defines generated API pages when a navigation tab or group references the
specification. A specification can also be loaded from a remote URL.
</Field>

<Field name="images/" type="directory">
Stores screenshots, diagrams, logos, favicons, and social-image backgrounds.
The directory name is a convention rather than a required Velu path.
</Field>

## Map source files to URLs

Page references are relative to the project root and omit `.mdx`:

```json
{
  "group": "Guides",
  "pages": ["quickstart", "guides/configuration", "guides/deployment"]
}
```

| Source file | Page reference | Default URL |
| --- | --- | --- |
| `index.mdx` | `index` | `/` |
| `quickstart.mdx` | `quickstart` | `/quickstart` |
| `guides/configuration.mdx` | `guides/configuration` | `/guides/configuration` |
| `guides/index.mdx` | `guides/index` | `/guides` |

Products, versions, and languages can add URL prefixes. The default value on
each switchable navigation axis remains unprefixed.

## Keep generated state out of source control

`velu dev` stores its dependency cache and version marker under `.velu/` in
the project. This directory is generated state, not authored documentation.
Add it to `.gitignore`:

```gitignore
.velu/
```

Commit `velu.json`, MDX pages, OpenAPI specifications, and project-owned
assets. Do not edit files inside `.velu/`.

## Use components without imports

Velu components are registered globally in MDX:

```mdx
<Callout type="tip">
  Preview the page before publishing it.
</Callout>
```

Component names and properties are case-sensitive. Do not import a built-in
component into an authored page.

## Verify the project structure

1. Run `velu dev` from the directory containing `velu.json`.
2. Open every new page through the sidebar and confirm its URL.
3. Confirm referenced local logos, favicons, images, and specifications load.
4. Run `velu validate` and resolve missing page or configuration errors.
