---
title: Markdown and text
description: Format prose, headings, links, lists, tables, and inline code in Velu MDX pages.
---

Velu compiles MDX with standard Markdown and GitHub Flavored Markdown. Use
Markdown for prose and built-in components for structured presentation.

## Format text

```md
Use **bold** for strong emphasis and _italics_ for light emphasis.

Use `inline code` for commands, file names, properties, and literal values.

~~Deprecated behavior~~ can be marked with strikethrough when the history is
important to the reader.
```

Use emphasis sparingly. Prefer direct sentences and descriptive headings over
formatting entire paragraphs for attention.

## Add headings

Start page sections at `##` because frontmatter renders the page's `h1`:

```md
## Configure the client

### Set the base URL

### Add authentication
```

Velu assigns heading IDs and builds the on-page table of contents from the
heading hierarchy. Keep levels sequential and do not skip from `##` to `####`.

## Add links

```md
[Validate your documentation](/getting-started/validate)

[Review component properties](/components/reference#field)

[Visit Velu](https://getvelu.com)
```

Use text that identifies the destination. For external links, include the
complete `https://` URL.

## Write lists

Use bullets when order does not matter:

```md
- Preview the page.
- Check its links.
- Review its alternative text.
```

Use numbers for independent ordered items. Use `Steps` when later actions
depend on earlier ones and each step needs supporting content.

```md
1. Create the page.
2. Add it to navigation.
3. Run validation.
```

GitHub-style task lists are supported:

```md
- [x] Write the page
- [ ] Review the rendered result
```

## Add tables

```md
| Command | Purpose |
| --- | --- |
| `velu dev` | Start the local preview. |
| `velu validate` | Check the project. |
```

Keep tables small enough to scan on narrow screens. Use `Field` elements when
each row needs a longer definition or required/default metadata.

## Use blockquotes

```md
> This behavior applies only to published sites.
```

Use a `Callout` instead when the content communicates a warning, risk,
constraint, or success condition.

## Mix Markdown and components

Components can contain Markdown content:

```mdx
<Callout type="warning">

Run `velu validate` before publishing.

</Callout>
```

Leave blank lines around block content inside a component. Component names and
properties are case-sensitive.

## Avoid raw HTML for layout

MDX treats HTML-like tags as JSX. Attributes and behavior can differ from
plain HTML, and custom styling is not part of the content contract. Prefer
Markdown and documented Velu components for portable pages.
