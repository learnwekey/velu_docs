---
title: How Velu works
description: Understand how the dashboard and docs-as-code workflows publish documentation.
---

A Velu project can be managed in the dashboard or through a connected Git
repository. The dashboard editor does not require Git. Connect a documentation
repository when you choose docs-as-code or need automation and self-updating
documentation.

## The two workflows

<Columns cols={2}>

<Card title="Dashboard editor" icon="panels-top-left">
Create, edit, preview, and publish documentation in the browser without a Git
repository or local tools.
</Card>

<Card title="Docs as code" icon="git-branch">
Store pages and configuration in Git, work locally, and deploy changes from a
connected repository. This workflow supports automation and self-updating
documentation.
</Card>

</Columns>

## Dashboard editor lifecycle

<Steps>

<Step title="Open the dashboard">
Sign in to Velu and open the dashboard. Connecting Git is optional.
</Step>

<Step title="Create or update content">
Use the web editor to write and preview the documentation.
</Step>

<Step title="Publish">
Use the editor's publish action, monitor its status, and verify the published
page.
</Step>

</Steps>

## Docs-as-code lifecycle

<Steps>

<Step title="Connect the repository">
From the dashboard, connect the Git repository that contains the documentation
project. This connection is required for docs-as-code.
</Step>

<Step title="Create or update content">
Edit Markdown and `velu.json` locally with an AI coding agent or editor.
</Step>

<Step title="Preview and check locally">
Run `velu run` to preview the site, then run `velu lint` to validate the
configuration and referenced pages.
</Step>

<Step title="Review and publish">
Review changes through your normal Git workflow. When an approved change
reaches the connected production branch, Velu builds and deploys the site.
</Step>

</Steps>

## Workflow responsibilities

| Dashboard editor | Docs as code |
| --- | --- |
| Does not require Git | Requires a connected Git repository |
| Edit and preview in the browser | Edit Markdown and configuration locally |
| Publish with the editor action | Preview and lint with local commands |
| Monitor publication in the dashboard | Review diffs, commits, and pull requests |
| Best for direct browser-based updates | Best for automation and self-updating docs |

<Callout type="note">

Docs-as-code users can run `velu build` to create a local static build in
`.velu-out/out`. Dashboard users can publish without installing the CLI.

</Callout>

## Why docs as code is recommended

The docs-as-code workflow gives documentation the same controls as software:

<ul style={{ listStyleType: "disc", paddingInlineStart: "1.5rem" }}>
  <li>Version history and rollback through Git.</li>
  <li>Pull-request review for content and configuration.</li>
  <li>Local preview before publication.</li>
  <li>Automated linting in local and CI workflows.</li>
  <li>Repository context for Codex, Claude Code, and other coding agents.</li>
</ul>

The dashboard editor remains useful for browser-based authoring and does not
require a repository. Choose docs-as-code when those Git controls or automated
documentation updates are part of your workflow.
